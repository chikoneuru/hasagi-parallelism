# Carbon intensity traces

Carbon is a derived proxy in HASAGI (energy × grid intensity); these are the
grid-intensity inputs. Three sources, in order of fidelity:

1. **Real, all 16 reported zones — ElectricityMaps API.** Fetch with
   `experiments/fetch_electricitymaps_traces.py` (set a token in
   `$ELECTRICITYMAPS_TOKEN`; historical `past-range` access needs a plan/academic
   licence that includes history). It writes one CSV per zone, named `<zone>_*.csv`,
   into `data_cache/real_traces/` (gitignored). `hasagi.energy.trace_schedule.load_zone_traces`
   then picks each up by the `<zone>_` filename prefix with zero code changes, and
   the experiments (`exp_realtrace_pareto`, `exp_realtrace_sensitivity`) consume them.
   The committed results used the 16-zone set over two seasons (2024-07 and 2024-01).

   ```bash
   export ELECTRICITYMAPS_TOKEN=...
   python -m experiments.fetch_electricitymaps_traces \
       --start 2024-07-01 --end 2024-07-15 --out data_cache/real_traces
   ```

2. **Synthetic per-zone fallback** — `hasagi.energy.carbon_trace.published_grid_trace(zone)`
   generates a multi-harmonic trace per zone from public annual statistics. Used
   when a real CSV is absent; outputs are labelled `synthetic-parametric` and must
   not be reported as measured. Cross-zone significance is computed over the REAL
   zones only.

3. **Synthetic solar** — `synthetic_solar.csv`, generated by
   `synthetic_solar_trace()` and committed for deterministic CI/unit tests (24 h
   cycle, 5-minute samples; low at solar noon, high overnight).

CSV schema (both real exports and plain drops): the ElectricityMaps Data Portal
columns (`Datetime (UTC)`, `Carbon Intensity gCO₂eq/kWh (LCA)`) are read directly;
a plain `timestamp,intensity_g_per_kwh` schema also works via `load_csv_trace`.

The 16 reported zones span ~30× intensity (NO/FR/BR clean → IN/PL/ZA dirty) and a
range of intra-day swing, which is what the carbon-shift analysis exploits.
